# IC3 子句排序讨论纪要（2026-03-08）

## 1. 当前共识
- 目标不是只比较速度，而是构建“可解释的评价方法”，说明某排序为何在某例子更优。
- `reset` 例子显示：`l1l2` 最优，但这只是单例，不足以指导全局策略。
- 需要通过 800+ 例子的预实验提炼“普遍现象”，再指导自适应排序。

## 2. reset 例子的关键发现（机制层）
- `l1l2` 的早期新子句更早落在 `intcounter` 的算术关系上，更接近该例子的核心约束。
- 因此状态空间更早被压缩，后续表现为：帧更小、求解调用更少、迭代更少。
- `l2l1/original` 前期更偏 `greycounter` 布尔字面，约束方向性较弱，收敛更慢。

## 3. 关键技术判断
- 现有 `l1l2/l2l1` 在代码里主要是字符串字典序（正/逆序），不是语义排序。
- 所以 `reset` 的优势很可能是“偶然命中”，不可直接泛化。

## 4. 预实验设计（800+ 样本）
1. 以“每个例子/性质”为样本，标签为三种排序赢家（最快者）。
2. 仅从早期过程提特征（前 N 个新子句或前 T 秒）：
   - 算术字面比例、布尔字面比例
   - 与 property 变量重合度
   - 变量簇命中（如 intcounter/greycounter）
   - 前期帧增长、前期 solver 调用速度
   - 差分约束/跨变量关系比例（方向性）
3. 做 winner-vs-loser 的配对差异统计（Wilcoxon + bootstrap + effect size）。
4. 用前期特征预测赢家（可解释模型），验证规律可用于在线决策。
5. 只保留运行时单次可观测特征，保证可部署。
6. 输出机制分型（例如 arithmetic-dominant / boolean-dominant / mixed）。
7. 离线仿真 selector 与固定策略对比，关注平均收益与最坏退化。

## 5. 后续实施方向（先不实现）
- 排序应从字符串序升级为语义序：
  - 算术关系优先
  - 与 property 相关变量优先
  - 在线根据 kept/removed 信号更新权重

  - 特征抽取 -> 统计检验 -> 规律分型 -> selector 离线仿真
